## 一、问题概述

### 1.1 问题描述

通过 Go 堆内存分析工具（`go tool pprof`）导出的 heap.txt 文件分析，发现项目存在严重的内存泄漏问题，核心表现为 **goroutine 泄漏**。

### 1.2 泄漏特征

| 指标 | 描述 |
|------|------|
| **泄漏类型** | Goroutine 泄漏（最严重） |
| **问题位置** | `doh/doh3.go` - HTTP/3 DNS 查询模块 |
| **影响组件** | quic-go QUIC Transport |
| **泄漏场景** | DNS-over-HTTPS (DoH3) 连接失败或超时后 |
| **症状** | Goroutine 数量持续增长，内存占用不断上升 |

## 二、根因分析

### 2.1 问题代码位置

```go
// doh/doh3.go - createH3Client 函数（修复前）
func createH3Client(dohurl string, dohip string) *h3ClientEntry {
    // 每次请求都创建新的 quic.Transport（底层 UDP socket）
    qt := &quic.Transport{Conn: udpConn}
    // ...
}
```

### 2.2 根因详解

#### 问题 1：Transport 重复创建

**原代码逻辑**：每次 DNS 查询都调用 `createH3Client()` 创建新的 `quic.Transport`

```go
// 问题代码
func doHTTP3ClientCached(msg *dns.Msg, ...) {
    entry := createH3Client(dohurl, dohip) // 每次请求创建新 Transport
    // ...
}
```

**问题**：
- 每次请求创建新的 `net.ListenUDP()` 监听 UDP 端口
- QUIC 连接失败后，底层 goroutine 未被正确终止
- `quic.Transport` 持有 UDP socket，长时间占用系统资源

#### 问题 2：缺少资源清理机制

**原代码缺失**：
- 无 `Close()` 方法释放 QUIC Transport
- 无缓存过期清理机制
- 无程序退出时的资源释放钩子
- 无原子状态标记防止重复关闭

#### 问题 3：goroutine 泄漏场景

```
DNS 查询请求
    ↓
创建 quic.Transport
    ↓
尝试 QUIC 连接 (DialEarly)
    ↓
连接失败/超时 → goroutine 被阻塞
    ↓
Transport 未被关闭 → goroutine 永远无法退出
    ↓
goroutine 泄漏 + UDP socket 泄漏
```

## 三、修复方案

### 3.1 核心设计

**策略**：复用 Transport + 完善生命周期管理

```
┌─────────────────────────────────────────────────────────────┐
│                    H3 Client Cache                          │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  Key: "dohurl|dohip"                                │   │
│  │  Value: h3ClientEntry                               │   │
│  │  ├── client: *http.Client                          │   │
│  │  ├── quicTransport: *quic.Transport               │   │
│  │  ├── h3Transport: *http3.Transport                │   │
│  │  ├── closed: atomic.Bool (防止重复关闭)            │   │
│  │  └── createdAt: time.Time (过期检查)              │   │
│  └─────────────────────────────────────────────────────┘   │
│                          ↓                                  │
│         同一 (dohurl, dohip) 复用同一个 Transport            │
└─────────────────────────────────────────────────────────────┘
```

### 3.3 详细修改内容

#### 3.3.1 h3ClientEntry 结构体增强

```go
type h3ClientEntry struct {
    client        *http.Client
    quicTransport *quic.Transport
    h3Transport   *http3.Transport
    mu            sync.Mutex
    closed        atomic.Bool  // 新增：原子标记，防止重复关闭
    createdAt     time.Time    // 新增：创建时间，用于超时检查
}
```

#### 3.3.2 新增 Close() 方法

```go
func (e *h3ClientEntry) Close() {
    if e.closed.Swap(true) {
        return // 已经关闭过了
    }
    e.mu.Lock()
    defer e.mu.Unlock()
    if e.client != nil {
        if t, ok := e.client.Transport.(*http3.Transport); ok {
            t.Close()
        }
    }
    if e.quicTransport != nil {
        e.quicTransport.Close()
    }
    if e.h3Transport != nil {
        e.h3Transport.Close()
    }
}
```

#### 3.3.3 新增过期检查方法

```go
func (e *h3ClientEntry) IsStale(maxAge time.Duration) bool {
    return time.Since(e.createdAt) > maxAge
}
```

#### 3.3.4 新增缓存清理机制

```go
// 启动定期清理器
func StartH3CacheCleaner(interval time.Duration, maxAge time.Duration) {
    h3CacheCleanerStop = make(chan struct{})
    h3CacheCleanerWG.Add(1)
    go func() {
        defer h3CacheCleanerWG.Done()
        ticker := time.NewTicker(interval)
        defer ticker.Stop()
        for {
            select {
            case <-ticker.C:
                CleanupStaleH3Clients(maxAge)
            case <-h3CacheCleanerStop:
                return
            }
        }
    }()
}

// 清理过期客户端
func CleanupStaleH3Clients(maxAge time.Duration) {
    h3ClientCache.Range(func(key, value interface{}) bool {
        entry := value.(*h3ClientEntry)
        if entry.IsStale(maxAge) {
            entry.Close()
            h3ClientCache.Delete(key)
        }
        return true
    })
}

// 程序退出时关闭所有缓存
func CloseH3ClientCache() {
    StopH3CacheCleaner()
    h3ClientCache.Range(func(key, value interface{}) bool {
        entry := value.(*h3ClientEntry)
        entry.Close()
        h3ClientCache.Delete(key)
        return true
    })
}
```

#### 3.3.5 getOrCreateH3Client 增强

```go
func getOrCreateH3Client(dohurl string, dohip string) *h3ClientEntry {
    cacheKey := dohurl + "|" + dohip
    if v, ok := h3ClientCache.Load(cacheKey); ok {
        entry := v.(*h3ClientEntry)
        // 检查是否已被关闭
        if entry.closed.Load() {
            entry.Close()
            h3ClientCache.Delete(cacheKey)
        } else {
            return entry
        }
    }
    // 创建新客户端或使用已有的
    entry := createH3Client(dohurl, dohip)
    if actual, loaded := h3ClientCache.LoadOrStore(cacheKey, entry); loaded {
        entry.Close()
        return actual.(*h3ClientEntry)
    }
    return entry
}
```

## 四、修复效果

### 4.1 修复前后对比

| 指标 | 修复前 | 修复后 |
|------|--------|--------|
| Transport 创建频率 | 每次请求 | 首次创建，长期复用 |
| Goroutine 泄漏 | 严重 | 无 |
| UDP Socket 泄漏 | 严重 | 无 |
| 资源清理 | 无 | 定期清理 + 退出时清理 |
| 请求超时 | 无 | 10 秒超时 |

### 4.2 修复原理

#### 原理 1：Transport 复用

```
修复前：
请求1 → 创建Transport1 → 连接失败 → goroutine泄漏
请求2 → 创建Transport2 → 连接失败 → goroutine泄漏
请求3 → 创建Transport3 → 连接成功 → 成功

修复后：
请求1 → 创建Transport1 → 连接成功 → Transport1 复用
请求2 → 使用Transport1 → 连接成功 → Transport1 复用
请求3 → 使用Transport1 → 连接成功 → Transport1 复用
```

#### 原理 2：原子状态标记

```go
// 使用 atomic.Bool 确保只关闭一次
func (e *h3ClientEntry) Close() {
    if e.closed.Swap(true) {  // 原子操作
        return // 已关闭，直接返回
    }
    // 执行关闭逻辑...
}
```

#### 原理 3：定期清理

```
时间线：
T0: 启动程序，创建 H3 客户端
T5min: 清理器检查，无过期客户端
T10min: 清理器检查，无过期客户端
...
T30min+: 长时间未使用的客户端被清理
```

## 八、总结

### 8.1 问题根因

HTTP/3 DNS 查询模块每次请求都创建新的 `quic.Transport`，连接失败后 goroutine 未被正确终止，导致严重的 goroutine 泄漏。

### 8.2 解决方案

1. **复用 Transport**：使用 `sync.Map` 缓存 H3 客户端
2. **完善生命周期**：添加 `Close()` 方法和退出钩子
3. **定期清理**：启动后台 goroutine 清理过期客户端
4. **超时控制**：设置请求超时防止永久阻塞

### 8.3 效果

- ✅ Goroutine 泄漏已消除
- ✅ UDP Socket 泄漏已消除
- ✅ 资源得到有效复用
- ✅ 程序退出时资源正确释放
